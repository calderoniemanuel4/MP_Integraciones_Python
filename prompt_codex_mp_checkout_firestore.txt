Actúa como arquitecto de software y desarrollador senior Python especializado en FastAPI, Cloud Firestore, APIs REST, webhooks e integraciones de pagos con Mercado Pago Checkout Pro.

Quiero que desarrolles un proyecto funcional de prueba para integrar Mercado Pago Checkout Pro usando Python, FastAPI y Cloud Firestore como única base de datos.

No usar SQL, SQLite, SQLAlchemy, SQLModel ni bases de datos relacionales.

## Objetivo general

Crear una aplicación que permita:

1. Crear una orden interna.
2. Guardar la orden en Cloud Firestore.
3. Crear una preferencia de pago mediante la API de Mercado Pago.
4. Mostrar un HTML local con un botón de pago.
5. Redirigir al comprador a Mercado Pago Checkout Pro.
6. Recibir notificaciones mediante un webhook público alojado en FastAPI Cloud.
7. Validar la firma de los webhooks de Mercado Pago.
8. Consultar el pago real mediante la API de Mercado Pago.
9. Guardar y actualizar pagos en Firestore.
10. Actualizar el estado de la orden relacionada.
11. Mostrar una página HTML de resultado mediante back URLs públicas.
12. Implementar idempotencia para webhooks, pagos y órdenes.
13. Detectar diferencias de monto o moneda.
14. Registrar alertas de conciliación.
15. Dejar una arquitectura preparada para incorporar posteriormente OAuth Partner, múltiples vendedores, reportes financieros, movimientos de billetera, Google Sheets, Looker Studio, Cloud Tasks, Pub/Sub y automatizaciones.

## Regla crítica sobre transacciones

No hacer llamadas HTTP a Mercado Pago dentro de una transacción de Firestore.

Flujo correcto:

1. Recibir webhook.
2. Validar firma.
3. Consultar pago en Mercado Pago.
4. Validar respuesta.
5. Iniciar transacción Firestore.
6. Leer estado actual del evento y de la orden.
7. Aplicar cambios atómicos.
8. Confirmar transacción.
9. Responder.

Las transacciones Firestore pueden reintentarse, por lo que no deben contener efectos externos.

- No exponer credenciales.
- No usar credenciales reales en tests.
- No escribir desde tests al proyecto Firestore real.
- No aprobar órdenes con datos de la back URL solamente.
- No actualizar pagos usando únicamente el payload del webhook.
- No usar precios del navegador como fuente definitiva.
- No usar float para dinero.
- No hacer llamadas HTTP dentro de transacciones.
- No usar IDs automáticos donde se requiere idempotencia.
- No usar SQL.
- No agregar complejidad innecesaria.
- No dejar métodos vacíos salvo extensiones futuras claramente documentadas.
- No utilizar código pseudofuncional.
- No crear un nuevo cliente Firestore por request.
- No permitir acceso directo del frontend a Firestore.

El proyecto estará completo cuando:

- GET /health responda correctamente.
- Se pueda crear una orden en Firestore.
- El ID del documento sea el order_id.
- external_reference sea igual a order_id.
- Se pueda crear una preferencia.
- Se guarden preference_id y URLs de Checkout.
- El HTML local pueda solicitar una preferencia.
- El navegador pueda redirigir a Checkout Pro.
- Las back URLs puedan consultar y mostrar el pago real.
- El webhook pueda recibir eventos.
- La firma pueda validarse.
